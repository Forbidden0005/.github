# Security Policy

## Reporting Vulnerabilities

Do not report suspected vulnerabilities in public issues, discussions, or pull requests.

Use GitHub private vulnerability reporting when it is available for the affected repository. If private reporting is not available, use a private channel already associated with the project or maintainer.

Do not put secrets, credentials, tokens, exploit code, proof-of-concept payloads, or step-by-step exploitation details in public GitHub surfaces.

For non-sensitive reliability, maintenance, or usability problems, use the normal issue templates.

## Scope

This policy applies to Forbidden0005 repositories that inherit these default community health files unless a repository-specific `SECURITY.md` overrides it.

## Expectations

Reports should be specific enough to reproduce and triage safely, but public reports must stay free of sensitive material. I will prioritize credible, reproducible reports and may close public issues that expose secrets or actionable exploit detail.
